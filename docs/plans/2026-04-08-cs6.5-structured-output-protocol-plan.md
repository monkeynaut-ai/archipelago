> **Upstream status (CS6):** CS6 is complete. It shipped as Option A — `ReviewerPayload` in `models.py` plus four role-spec YAMLs. CS6.5 depends on no new CS6 type beyond `ReviewerPayload`; the other three agent payload types (`ImplementationTask`, `DispatcherOutput`, `IntegratorOutput`) are CS5 domain types that already exist in `models.py`. The old `PlannerOutput`/`NewDispatcherOutput`/`IntegratorAgentOutput` wrappers planned in pre-Option-A CS6 were intentionally not built — CS7 will call `build_outcome_schema(...)` on the domain types directly.

**Goal:** Establish the structured-output protocol contract so CS7/CS8 agent handlers can invoke `claude --json-schema`, receive typed agent outcomes, and route them (success / clarification / permission / failure) without parsing free-text markers.

**Architecture:** Agent Foundry gains a `--json-schema`-aware adapter path — it detects Claude Code's synthetic `StructuredOutput` tool call, captures the payload as a new protocol message, and surfaces it via `TurnResult.structured_output`. A small schema-flattening helper converts Pydantic-generated JSON Schema into a Claude-Code-compatible form (inlines `$ref`, strips OpenAPI `discriminator`). Archipelago defines a generic `AgentTurnEnvelope[T]` discriminated-union envelope with four variants (success, clarification_needed, permission_needed, failed). No CS7/CS8 agent handlers are written in this change set — CS6.5 only delivers the protocol machinery and the typed envelope. The existing text-marker protocol is left intact and untouched as a fallback; CS11 will delete it once all agents are on the new path.

**Tech Stack:** Python 3.14, Pydantic v2, PDM, pytest, `claude` CLI version 2.x.

## Conversation design decisions (load-bearing context for implementers)

These were locked in during the CS6 planning session. Do not re-litigate.

1. **Four outcome kinds, no more**: `success` (carries typed payload), `clarification_needed`, `permission_needed`, `failed`. Terminal external failures (timeout, crash, rate-limit) are handled by the adapter, not by StructuredOutput.
2. **`FailureOutcome` uses `reason: str`** — no `FailureCategory` enum. Archipelago will promote patterns to an enum later if field data justifies it.
3. **One unified `TurnOutcomeMessage` message type** over the wire, with the discriminated outcome nested as a field. Matches the Pydantic envelope shape.
4. **Schema flattening is required** — verified empirically in the planning session. Claude Code's `--json-schema` enforcement:
   - **Silently disables** when the schema contains the OpenAPI `discriminator` keyword (Pydantic emits this by default for `Field(discriminator="kind")`). `subtype: success`, `structured_output` key absent — most dangerous failure mode.
   - **Fails with `error_max_structured_output_retries`** when the schema contains `$defs` + `$ref` (Pydantic emits these for every nested `BaseModel`). Visible failure, wasted turn.
   - **Works correctly** when the schema is fully inlined (no `$defs`, no `$ref`) and `discriminator` is stripped. Nested `oneOf` with `const` discriminators at multiple levels works on first try.
5. **Envelope lives in Archipelago**, not Agent Foundry. Agent Foundry's adapter carries `structured_output: dict[str, Any]` generically; Archipelago validates the dict into `AgentTurnEnvelope[T]`. This preserves Agent Foundry as a domain-agnostic framework.
6. **All agents use `--json-schema`** — no two-class split between data-producing and file-producing agents. File-producing agents' metadata (e.g. `files_modified: list[str]`) is a legitimate success payload shape. CS8 will define those output models.
7. **Pydantic v2 handles the generic envelope + discriminated union + generic variant correctly** — empirically verified during the CS6.5 planning session. `AgentTurnEnvelope[TrivialPayload]` parametrizes at module load, generates a schema (with `$defs`/`$ref` that the inliner strips), validates all four variants into typed instances (including `SuccessOutcome[TrivialPayload]` as the parametrized variant), and round-trips cleanly. No need for fallback designs (non-generic envelope + dict payload + subclass factories).
8. **`Literal[TurnOutcomeKind.VARIANT]` is required, not optional, on each outcome class** — empirically verified. Pydantic v2.12 rejects bare StrEnum-typed discriminator fields with `PydanticUserError: Model 'X' needs field 'kind' to be of type 'Literal'`. The `Literal[SomeEnum.VARIANT]` form is the exact fallback clause that CLAUDE.md's data model conventions permit. The enum remains the source of truth; the Literal wrapper exists only to satisfy Pydantic.

  ```python
  """Schema transformation utilities for Claude Code --json-schema compatibility.

  Claude Code's --json-schema flag has two limitations that Pydantic's default
  JSON Schema output violates:

  1. It cannot resolve `$ref` references. Pydantic generates `$defs` + `$ref`
     for every nested BaseModel. Leaving them in produces
     `error_max_structured_output_retries` after 5 failed attempts.

  2. It silently disables schema enforcement when the OpenAPI `discriminator`
     keyword is present. Pydantic emits this for any `Field(discriminator="...")`
     on a discriminated union. The subtype comes back as `success` but
     `structured_output` is absent — the most dangerous failure mode because it
     looks like everything worked.

  This module's `to_claude_code_schema` function transforms a Pydantic model's
  schema into a standalone, reference-free form that Claude Code accepts.
  Discriminated unions remain expressible via plain `oneOf` with `const` kind
  discriminators — empirically verified to work correctly, including nested
  cases.
  """

  from copy import deepcopy
  from typing import Any

  from pydantic import BaseModel


  def _resolve_ref(ref: str, defs: dict[str, Any]) -> dict[str, Any]:
      """Resolve a '#/$defs/Name' reference against a $defs dict."""
      prefix = "#/$defs/"
      if not ref.startswith(prefix):
          raise ValueError(f"unsupported $ref shape: {ref!r}")
      name = ref[len(prefix):]
      if name not in defs:
          raise KeyError(f"missing $def: {name!r}")
      return defs[name]


  def _inline(node: Any, defs: dict[str, Any]) -> Any:
      """Walk a JSON Schema node, inlining $refs and stripping discriminator / $defs keys."""
      if isinstance(node, dict):
          # A bare $ref node: replace with the referenced definition (recursively inlined).
          if "$ref" in node and len(node) == 1:
              return _inline(deepcopy(_resolve_ref(node["$ref"], defs)), defs)
          result: dict[str, Any] = {}
          for key, value in node.items():
              if key == "discriminator":
                  continue  # OpenAPI extension — breaks Claude Code's schema enforcement
              if key == "$defs":
                  continue  # inlined; drop from output
              result[key] = _inline(value, defs)
          return result
      if isinstance(node, list):
          return [_inline(item, defs) for item in node]
      return node


  def to_claude_code_schema(model: type[BaseModel]) -> dict[str, Any]:
      """Convert a Pydantic model's JSON schema into a Claude-Code-compatible form.

      Transforms:
          - Inlines all $defs/$ref references (Claude Code cannot resolve them)
          - Strips the OpenAPI `discriminator` keyword (causes silent schema disable)

      The returned dict is a standalone JSON Schema document ready to pass to
      `claude --json-schema`.
      """
      raw = model.model_json_schema()
      defs = raw.get("$defs", {})
      return _inline(raw, defs)
  ```

### Task 4d: stderr-on-error — hybrid path, one status per turn

**Design rationale:** stderr is normally silent, but on failure the orchestrator needs to see the root cause. There are two distinct failure modes, each requiring a different integration point:

| Failure mode | `result` event fires? | Fix path |
|---|---|---|
| LLM turn ended with `is_error=True` | yes | Fold stderr tail into the existing `turn_complete` status's `detail` field inside `_map_event_to_protocol` |
| Process crashed before emitting any `result` (OOM, segfault, timeout, rate-limit 429) | no | After `proc.wait()` in `run_turn`, emit a synthetic `error` status with the stderr tail — but only if no terminal status was sent during the event loop |

The adapter must emit **exactly one terminal status per turn** regardless of which path fires. The existing adapter at `claude_code.py:176-188` already emits `status="turn_complete"` with `exit_code=1` on `is_error=True`; we extend that emission to carry stderr detail. The new post-wait path is the safety net for cases where no `result` event ever arrived.

**Plumbing constraint:** `_map_event_to_protocol` currently has no access to the stderr buffer (`stderr_lines` is local to `run_turn`). The fix is to pass a read-only accessor into `_map_event_to_protocol` as an optional argument, defaulting to `None` to keep existing callers unaffected.

  ```python
  """Generic turn-outcome envelope for Archipelago agents.

  Every agent running under Claude Code's --json-schema returns its turn result
  as an AgentTurnEnvelope[T], where T is the agent-specific success payload.
  The envelope's outcome field is a discriminated union of four variants:

      - SuccessOutcome[T]      — normal completion with typed payload
      - ClarificationOutcome   — agent needs human input to proceed
      - PermissionOutcome      — agent needs approval to proceed
      - FailureOutcome         — agent declares the task cannot be completed

  This envelope is the contract for all CS7/CS8 agent handlers. The schema is
  injected into `claude --json-schema` via build_outcome_schema (task 6).

  Design discussions:
      - docs/plans/2026-04-03-review-feedback-loop-design.md
      - CS6/CS6.5 planning conversation (summarized in the CS6.5 plan doc)

  Rationale notes:
      - FailureOutcome uses a free-form `reason: str` — no FailureCategory
        enum. Archipelago team will promote patterns to an enum later if field
        data justifies it.
      - Kind values use StrEnum (per project convention) but are wrapped with
        ``Literal[TurnOutcomeKind.VARIANT]`` on each outcome class. This is
        the project rule's explicit fallback clause: Pydantic v2.12 rejects
        bare StrEnum-typed discriminator fields with
        ``PydanticUserError: Model 'X' needs field 'kind' to be of type
        'Literal'``. Empirically verified in the CS6.5 planning session.
        The ``Literal[SomeEnum.VARIANT]`` form preserves the enum as the
        source of truth while satisfying Pydantic's discriminator
        requirement. See docs/plans/2026-04-08-cs6.5-structured-output-protocol-plan.md
        and the project CLAUDE.md data model conventions.
  """

  from enum import StrEnum
  from typing import Annotated, Generic, Literal, TypeVar

  from pydantic import BaseModel, Field

  T = TypeVar("T", bound=BaseModel)


  class TurnOutcomeKind(StrEnum):
      SUCCESS = "success"
      CLARIFICATION_NEEDED = "clarification_needed"
      PERMISSION_NEEDED = "permission_needed"
      FAILED = "failed"


  class SuccessOutcome(BaseModel, Generic[T]):
      kind: Literal[TurnOutcomeKind.SUCCESS] = TurnOutcomeKind.SUCCESS
      payload: T


  class ClarificationOutcome(BaseModel):
      kind: Literal[TurnOutcomeKind.CLARIFICATION_NEEDED] = TurnOutcomeKind.CLARIFICATION_NEEDED
      question: str = Field(description="What the agent needs to know to proceed")
      options: list[str] = Field(
          default_factory=list,
          description="Possible answers the human can pick from, if applicable",
      )
      blocking: bool = Field(
          default=True,
          description="Whether the agent must receive an answer before continuing",
      )


  class PermissionOutcome(BaseModel):
      kind: Literal[TurnOutcomeKind.PERMISSION_NEEDED] = TurnOutcomeKind.PERMISSION_NEEDED
      action: str = Field(description="Action the agent wants to take")
      risk_level: str = Field(description="Suggested taxonomy: low | medium | high")
      why_needed: str = Field(description="Why the action is necessary for the task")


  class FailureOutcome(BaseModel):
      kind: Literal[TurnOutcomeKind.FAILED] = TurnOutcomeKind.FAILED
      reason: str = Field(
          description="Why the agent cannot proceed and why clarification/permission will not help"
      )
      attempted_approaches: list[str] = Field(
          default_factory=list,
          description="If the agent tried multiple approaches before giving up, list them here",
      )


  class AgentTurnEnvelope(BaseModel, Generic[T]):
      """Top-level envelope wrapping the discriminated outcome.

      Wrapping the union in an object with a single `outcome` field is required
      because Claude Code's --json-schema enforcement requires the top-level
      schema to be an object type; top-level oneOf is silently ignored.
      """

      outcome: Annotated[
          SuccessOutcome[T] | ClarificationOutcome | PermissionOutcome | FailureOutcome,
          Field(discriminator="kind"),
      ]
  ```
